# FE 작업 지침

이 폴더는 LG Able Band 프론트엔드 작업의 중심이다.
다만 FE 화면만 그럴싸하게 만드는 것이 목표가 아니라, 실제 서비스 기능이 동작하도록 필요한 도메인과 함께 구현한다.

## 작업 범위

- FE 요청의 기본 수정 범위는 사용자가 요청한 기능과 직접 관련된 `FE/` 파일로 한정한다.
- 실제 기능 이해를 위해 `BE/`, `ML/`, `HW/`를 읽을 수 있지만, 사용자가 요청하지 않은 도메인 파일은 기본적으로 수정하지 않는다.
- FE 요청이라도 실제 기능 완성에 `BE/`, `ML/`, `HW/` 수정이 필요해 보이면 먼저 영향 범위와 이유를 설명하고, 사용자가 동의한 뒤 수정한다.
- API 계약, 저장 구조, 웨어러블 연동, 하드웨어 이벤트가 필요한 기능은 FE 안에서 mock/hardcoding으로만 끝내지 않는다.
- 단순 UI 정리, 문구 수정, 화면 상태 개선처럼 FE 내부에서 끝나는 작업은 `FE/` 안에서 처리한다.
- `FE/app`은 사용자 모바일 앱, `FE/wearable`은 밴드/웨어러블 화면이다. 알림, 긴급 요청, 접근성, UWB처럼 양쪽에 영향을 주는 기능은 둘 다 확인한다.
- ML에서 전달되는 프론트 연동부, 팀원이 작업 중일 수 있는 화면/API 연결부, 서버 불안정 우회 코드는 사용자가 명시적으로 요청하지 않는 한 수정하지 않는다.

## 구현 원칙

- 사용자가 누르고, 입력하고, 결과를 보고, 실패했을 때 이유를 알 수 있는 완성된 흐름을 만든다.
- 변경 파일 수를 최소화하고, 요청한 기능과 직접 관련 없는 리팩터링/정리/문서 갱신은 하지 않는다.
- 별도 요청이 없으면 UI/UX를 크게 리디자인하지 않고, 사용자가 지적한 기능/상태 문제를 해결하는 데 필요한 최소 화면 수정만 한다.
- API가 아직 없으면 mock data로 화면을 먼저 만들 수 있지만, 실제 연동이 필요한 기능이면 서비스 레이어와 백엔드 계약까지 이어지게 한다.
- mock data는 실제 API로 교체하기 쉽도록 서비스 레이어에 격리한다.
- 하드코딩된 수치, 상태, 위치, 알림 문구가 실제 데이터처럼 보이면 우선 의심하고 데이터 출처를 확인한다.
- 접근성 사용자가 이해할 수 있도록 음성, 진동, 화면, 텍스트, 색상 보조 표현을 고려한다.
- 로딩, 빈 데이터, 오류, 권한 없음 상태를 함께 구현한다.

## 실행과 검증

`FE/app`과 `FE/wearable`은 각각 Vite React 앱이다.

- 개발 서버: 각 앱 폴더에서 `npm run dev`
- 테스트: 각 앱 폴더에서 `npm run test`
- 린트: 각 앱 폴더에서 `npm run lint`
- 빌드: 각 앱 폴더에서 `npm run build`

기능 구현 후에는 가능한 범위에서 테스트, 빌드, 실제 브라우저 화면 확인을 함께 수행한다.
