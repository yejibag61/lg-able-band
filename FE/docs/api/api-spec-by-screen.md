# LG Able Band 화면별 API 명세서

상태: FE-BE 협의용 화면 기준 초안  
원본 기준: `최종_LG_Able_Band_개발산출물_260609.docx`  
상세 API 초안: `docs/api/api-spec-draft.md`  
화면 기준 문서: `docs/frontend/fe-screens.md`, `docs/frontend/screen-api-map.md`

이 문서는 FE 개발자가 화면을 만들 때 필요한 API와 데이터를 빠르게 확인하기 위한 화면별 명세서다.
요청/응답 JSON 예시, 공통 Enum, 에러 응답 형식은 `docs/api/api-spec-draft.md`를 우선 확인한다.

## 읽는 방법

- `구현 기능`: 화면에서 사용자가 실제로 할 수 있어야 하는 행동
- `API`: 해당 기능을 위해 FE가 호출하거나 참고할 백엔드 API
- `데이터`: FE 화면에서 state, props, mock data, API 응답으로 관리할 값
- `백엔드가 프론트에 보내는 데이터`: 백엔드 응답에 포함되어야 하는 핵심 필드와 이유

## 공통 결정값

| 항목 | 산출물 기준 결정값 | 비고 |
| --- | --- | --- |
| 로그인 역할 | `USER`, `GUARDIAN` | 사용자는 메인 앱 홈으로, 보호자는 보호자 위험 알림/이력 중심 화면으로 진입 |
| 장애 유형 | `VISUAL`, `HEARING` | 시각장애인, 청각장애인만 사용 |
| 홈 데이터 | `GET /api/app/home` 우선 | 홈 첫 화면은 한 번에 조회하는 구조를 우선 |
| 알림 유형 | `LIFE`, `DANGER`, `EMERGENCY`, `LOCATION` | 생활, 위험, 긴급, 위치 안내 |
| 위험도 | `LOW`, `MEDIUM`, `HIGH`, `CRITICAL` | 위험/긴급 알림 강조 기준 |
| 다시 듣기 | FE Web Speech API 우선 | 백엔드 TTS URL은 확장 옵션 |
| UWB 거리 갱신 | MVP는 polling 우선 | 이후 WebSocket/SSE 확장 가능 |

## 공통 데이터 중복 방지 원칙

반복되는 데이터는 화면마다 새 구조로 만들지 않고 공통 객체를 재사용한다.

| 공통 객체 | 포함 데이터 | 사용하는 화면 |
| --- | --- | --- |
| `AccountSummary` | `accountId`, `role`, `name`, `email` | 로그인, 메뉴 |
| `UserSummary` | `userId`, `name`, `accessibilityType`, `notificationPrefs` | 홈, 접근성 설정 |
| `GuardianSummary` | `guardianId`, `name`, `phone`, `relationship`, `isPrimary`, `status` | 홈, 보호자 연결, 긴급 요청 |
| `AlertSummary` | `alertId`, `type`, `severity`, `title`, `message`, `deviceName`, `occurredAt`, `status` | 홈, 알림 목록, 이력, 웨어러블 |
| `DeviceSummary` | `deviceId`, `name`, `type`, `connectionStatus`, `uwbSupported`, `lastEventAt` | 홈, 기기 연동, UWB |
| `UwbSessionSummary` | `sessionId`, `targetDeviceId`, `targetDeviceName`, `distanceM`, `confidence`, `navigationStatus`, `vibrationPattern`, `voiceGuide` | UWB, 웨어러블 UWB |

- 홈 API는 위 공통 객체의 요약본을 조합해서 내려준다.
- 상세 화면은 요약 객체의 ID를 기준으로 상세 API를 다시 조회한다.
- 같은 의미의 필드는 `id`/`name`/`status`처럼 화면마다 다른 이름으로 만들지 않는다.

## 0. 로그인/역할 선택 화면

### 0-1 구현 기능

| 기능 | 설명 |
| --- | --- |
| 계정 유형 선택 | 사용자 또는 보호자 계정 유형을 선택 |
| 이메일/비밀번호 입력 | 로그인에 필요한 기본 정보 입력 |
| 사용자 로그인 | 사용자 계정으로 로그인 후 홈 화면으로 이동 |
| 보호자 로그인 | 보호자 계정으로 로그인 후 보호자 알림/이력 화면으로 이동 |
| 로그인 실패 처리 | 계정 없음, 비밀번호 오류, 권한 없음 안내 |
| 세션 저장 | 로그인 성공 후 access token과 role 저장 |

### 0-2 API

#### 로그인 실행 흐름

| API | 필요 여부 | 설명 |
| --- | --- | --- |
| `POST /api/auth/login` | 필수 | 이메일, 비밀번호, 역할로 로그인 |

### 0-3 데이터

| 데이터 | 처리 방식 |
| --- | --- |
| 로그인 역할 | 프론트 state로 `USER` 또는 `GUARDIAN` 관리 |
| 이메일 | 프론트 입력 state로 관리 |
| 비밀번호 | 프론트 입력 state로 관리 |
| access token | 로그인 성공 응답으로 저장 |
| 로그인 계정 요약 | 백엔드 응답의 `account`로 관리 |
| 로그인 실패 메시지 | 백엔드 에러 또는 프론트 고정 문구로 표시 |

### 백엔드가 프론트에 보내는 데이터

| 데이터 | 필요한 이유 |
| --- | --- |
| access token | 이후 인증 API 호출 |
| role | 사용자/보호자 화면 분기 |
| account ID | 로그인 계정 식별 |
| 사용자 ID | role이 `USER`일 때 홈/설정/알림 조회 기준 |
| 보호자 ID | role이 `GUARDIAN`일 때 보호자 화면 조회 기준 |
| 이름 | 상단 프로필 또는 메뉴 표시 |
| 이메일 | 계정 정보 표시 |
| 장애 유형 | role이 `USER`일 때 접근성 UI 적용 |
| 연결된 사용자 ID | role이 `GUARDIAN`일 때 확인할 사용자 식별 |

- 로그인 요청에는 `role`을 포함해 사용자/보호자 계정을 명확히 구분한다.
- 보호자는 접근성 설정 대상이 아니므로 `accessibilityType`을 받지 않는다.

## 1. 홈 화면

### 1-1 구현 기능

| 기능 | 설명 |
| --- | --- |
| 홈 진입 | 로그인 후 사용자가 처음 보는 기본 화면 표시 |
| 오늘의 상태 확인 | 안전/주의/위험/긴급 상태를 카드로 표시 |
| 최근 알림 확인 | 최근 생활/위험/긴급 알림 3개 정도를 홈에 표시 |
| 알림 상세 이동 | 최근 알림을 선택하면 알림 상세/다시 듣기 화면으로 이동 |
| 긴급 도움 요청 | 큰 SOS 버튼으로 보호자 또는 백엔드에 긴급 요청 생성 |
| 기기 상태 확인 | 연결 기기 수, 오류 기기 수, 웨어러블 연결 상태 표시 |
| UWB 위치 안내 진입 | 위치 안내 가능한 기기가 있으면 UWB 화면으로 이동 |
| 빈 상태 처리 | 알림 없음, 기기 없음, 보호자 없음 상태 안내 |

### 1-2 API

#### 홈 데이터 조회 흐름

| API | 필요 여부 | 설명 |
| --- | --- | --- |
| `GET /api/app/home` | 필수 | 홈 화면에 필요한 사용자, 상태, 최근 알림, 기기 요약을 한 번에 조회 |
| `GET /api/alerts?limit=5` | 대체 | `GET /api/app/home`에서 최근 알림을 받지 못할 경우만 사용 |
| `GET /api/devices` | 대체 | `GET /api/app/home`에서 기기 요약을 받지 못할 경우만 사용 |
| `GET /api/uwb/targets` | 선택 | 홈에서 바로 UWB 대상 목록을 펼쳐 보여줄 경우만 사용 |
| `POST /api/emergency-requests` | 액션 | 긴급 도움 요청 버튼 클릭 시 호출 |

### 1-3 데이터

| 데이터 | 처리 방식 |
| --- | --- |
| 사용자 이름 | 백엔드 응답 또는 mock data로 관리 |
| 현재 공간명 | 백엔드 응답 또는 프론트 고정값 |
| 오늘 안전 상태 | 백엔드 응답의 `safetyStatus`로 관리 |
| 최근 알림 목록 | 백엔드 응답 또는 mock alert 배열로 관리 |
| 연결 기기 요약 | 백엔드 응답의 `deviceSummary`로 관리 |
| 보호자 연결 요약 | 백엔드 응답의 `guardianSummary`로 관리 |
| 긴급 요청 상태 | 프론트 state로 `idle`, `sending`, `success`, `error` 관리 |
| 카드별 로딩/오류 | 프론트 state로 관리 |

### 백엔드가 프론트에 보내는 데이터

| 데이터 | 필요한 이유 |
| --- | --- |
| 사용자 이름 | 홈 상단 인사말 또는 사용자 식별 표시 |
| 현재 공간명 | `우리집`, `Able Band 홈` 같은 공간 표시 |
| 안전 상태 레벨 | 홈의 가장 중요한 상태 카드 표시 |
| 안전 상태 문구 | 사용자가 현재 상황을 바로 이해하도록 표시 |
| 마지막 업데이트 시간 | 정보 신뢰도와 최신성 표시 |
| 최근 알림 목록 | 홈에서 놓친 알림을 빠르게 확인 |
| 연결 기기 수 | 서비스 연결 상태 요약 표시 |
| 오류/주의 기기 수 | 사용자가 확인해야 할 기기 상태 표시 |
| 웨어러블 연결 여부 | Able Band 연결 상태 표시 |
| UWB 가능 기기 수 | 위치 안내 카드 표시 여부 판단 |
| 주 보호자 이름 | 긴급 요청 가능 여부와 보호자 연결 상태 표시 |
| 긴급 요청 가능 여부 | SOS 버튼 활성/비활성 판단 |

- 홈 화면은 `GET /api/app/home` 하나로 구성하는 것을 우선 결정값으로 둔다.
- `guardianSummary`는 홈 요약에 포함하고, 보호자 전체 목록은 보호자 관리 화면에서만 `GET /api/guardians`로 조회한다.
- 알림과 기기 상세가 필요한 경우에만 개별 API를 추가 호출한다.
- 홈에서 API 일부가 실패해도 전체 화면을 막지 않고 해당 카드에만 재시도 버튼을 보여준다.

## 2. 알림 목록 화면

### 2-1 구현 기능

| 기능 | 설명 |
| --- | --- |
| 알림 목록 조회 | 생활/위험/긴급 알림을 최신순으로 표시 |
| 알림 유형 필터 | 생활, 위험, 긴급 알림을 구분해서 조회 |
| 확인 상태 필터 | 미확인/확인 완료 알림을 구분 |
| 알림 확인 | 사용자가 알림을 확인 완료 처리 |
| 다시 듣기 | 알림 내용을 음성 또는 텍스트 안내로 다시 확인 |
| 상세 보기 | 알림 상세/다시 듣기 화면으로 이동 |
| 위험 알림 강조 | 위험/긴급 알림을 카드 색상, 라벨, 문구로 강조 |
| 빈 알림 처리 | 알림이 없을 때 안내 문구 표시 |

### 2-2 API

#### 알림 목록 조회와 응답 흐름

| API | 필요 여부 | 설명 |
| --- | --- | --- |
| `GET /api/alerts?type=&status=&limit=20` | 필수 | 알림 목록 조회 |
| `GET /api/alerts/{alertId}` | 선택 | 상세 이동 전에 최신 상세 데이터를 조회할 때 사용 |
| `POST /api/alerts/{alertId}/confirm` | 액션 | 알림 확인 완료 처리 |
| `POST /api/alerts/{alertId}/replay` | 액션 | 다시 듣기 실행 또는 다시 듣기 이력 저장 |

### 2-3 데이터

| 데이터 | 처리 방식 |
| --- | --- |
| 알림 목록 | 백엔드 응답의 `items` 배열로 관리 |
| 현재 필터 | 프론트 state로 `type`, `status` 관리 |
| 선택한 알림 ID | 상세 이동 시 route param 또는 state로 전달 |
| 확인 처리 상태 | 프론트 state로 로딩/성공/실패 관리 |
| 다시 듣기 상태 | 프론트 state로 재생 중/완료/실패 관리 |
| 빈 알림 문구 | 프론트 고정값 |

### 백엔드가 프론트에 보내는 데이터

| 데이터 | 필요한 이유 |
| --- | --- |
| 알림 고유 ID | 상세 조회, 확인, 다시 듣기 API 요청 |
| 알림 유형 | 생활/위험/긴급 라벨과 필터 표시 |
| 위험도 | 카드 강조 수준 판단 |
| 제목 | 알림 카드 첫 줄 표시 |
| 짧은 메시지 | 알림 카드 설명 표시 |
| 기기명 | 어떤 기기에서 발생했는지 표시 |
| 발생 위치 | 위험/위치성 알림의 맥락 표시 |
| 발생 시간 | 최신순 정렬과 사용자 인지 |
| 확인 상태 | 미확인/확인 완료 UI 표시 |
| 보호자 알림 필요 여부 | 위험 알림 후속 흐름 판단 |

- 알림 목록은 최신순을 기본으로 한다.
- 위험/긴급 알림은 색상만으로 구분하지 않고 `위험`, `긴급` 텍스트 라벨을 함께 표시한다.
- 실시간성은 MVP에서는 polling으로 시작하고, 이후 WebSocket 또는 SSE를 협의할 수 있다.

| 다시 듣기 상태 | 프론트 state로 관리 |
| 진동 상태 | 백엔드 응답 또는 프론트 규칙으로 표시 |

## 백엔드와 우선 협의할 항목

협의 항목은 질문이 아니라 산출물 기준의 FE 우선 결정안으로 정리한다.
백엔드 구현 상황에 따라 변경될 수 있지만, FE는 아래 값을 기준으로 mock data와 화면을 먼저 만든다.

| 우선순위 | 협의 항목 | 산출물 기준 결정값 | 이유 |
| --- | --- | --- | --- |
| 1 | 로그인 역할 | `USER`, `GUARDIAN` | 사용자 앱과 보호자 확인 흐름을 로그인 시점에 분기 |
| 2 | 장애 유형 | `VISUAL`, `HEARING` | 산출물 기준 대상은 시각장애인, 청각장애인 |
| 3 | `GET /api/app/home` 응답 구조 | 홈 요약 API 1개로 우선 구성 | 홈 화면과 시연 첫 화면에 필요 |
| 4 | `safetyStatus.level` 값 목록 | `SAFE`, `CAUTION`, `DANGER`, `EMERGENCY` | 홈 상태 카드와 위험 강조 기준 |
| 5 | 알림 유형 | `LIFE`, `DANGER`, `EMERGENCY`, `LOCATION` | 생활 신호, 위험 신호, 긴급 요청, 위치 안내 구분 |
| 6 | 위험도 값 | `LOW`, `MEDIUM`, `HIGH`, `CRITICAL` | 위험/긴급 알림 강조 기준 |
| 7 | 알림 실시간성 방식 | MVP는 polling 우선 | WebSocket/SSE 없이도 병렬 개발 가능 |
| 8 | 다시 듣기 방식 | FE Web Speech API 우선 | 백엔드 TTS 없이도 접근성 흐름 시연 가능 |
| 9 | 긴급 요청 실패 사유 코드 | `NO_GUARDIAN`, `DELIVERY_FAILED`, `UNAUTHORIZED`, `SERVER_ERROR` | 보호자 없음, 전송 실패, 권한 없음 처리 |
| 10 | UWB 거리 갱신 방식 | MVP는 polling 우선 | 이후 WebSocket/SSE 확장 가능 |
| 11 | 보호자 권한 정책 | `GUARDIAN` role 로그인 후 연결된 사용자 알림만 조회 | 보호자 화면 접근 범위 명확화 |
